## Important (must fix before code)

- **I-1 — The gather→single-cardMD1 extraction is unspecified and the cited reuse (`singleSigReadbackCards`) is wrong for the supply.** Spec §2/§3 say "Gather the supplied md1 over NFC via `bundleGatherFlow` → the single `cardMD1`," and the recon (line 12) says to "clone `singleSigReadbackCards`." But `bundleGatherFlow` (`gui/bundle_flow.go:95`) accumulates *arbitrarily many* md1 AND mk1 cards (`gui/bundle.go:127-252`) — it does NOT enforce "exactly one md1, zero mk1." And `singleSigReadbackCards` (`gui/singlesig_verify.go:23-42`) is unusable here: it *requires* both an mk1 AND an md1 (`if len(mk1)==0 || len(md1)==0 → false`, `:38`), the opposite of a multisig supply (one md1, zero mk1). The spec must define a NET-NEW extraction with explicit failure modes: 0 md1 → refuse; ≥2 md1 → refuse (ambiguous); stray mk1 cards present → refuse or ignore (state which). As written, an operator scanning the supply has no deterministic single-md1 contract, and the named reuse target rejects the valid case. Add this helper to the §3 NET-NEW list and to the §5 acceptance gate.

- **I-2 — The canonical-pair compare as written (`cc==k.Xpub[0:32]`) is type-incorrect and the spec presents it as the load-bearing D14 rule.** `decodeXpubBytes` returns `chainCode [32]byte`, `compressedPubkey [33]byte` (fixed arrays, `gui/singlesig_derive.go:99`), while `k.Xpub[0:32]` / `k.Xpub[32:65]` are `[]byte` slices (`md/expand.go:62`, layout confirmed by existing `gui/md1_expand.go:65-66`). In Go you cannot `==` a `[N]byte` against a `[]byte` — it does not compile. The correct form is `bytes.Equal(cc[:], k.Xpub[0:32]) && bytes.Equal(pk[:], k.Xpub[32:65])` (or `cc == [32]byte(k.Xpub[0:32])`, available on go 1.25). Because §2/§6-I1/§7 elevate this exact expression to *the* wrong-wallet guard, the spec's normative pseudocode for the single Critical-class primitive must be corrected so the implementer codes the compare against the real return types — not left as a non-compiling form that a reader could "fix" in a subtly wrong way (e.g. comparing only 32 of 33 pubkey bytes, or forgetting the trailing byte of `[32:65]`).

- **I-3 — The multisig restore-doc render path is net-new but unenumerated; the existing `restoreDocFlow` cannot be reused as cited.** §2 says addresses come "via `expandedToDescriptor` → `address.Receive/Change`." But the shipped `restoreDocFlow` (`gui/singlesig_restore.go:118`) takes single-sig scalars (`xpub, masterFP, parentFP, script, path`) and builds a *single-sig* descriptor internally (`:65,:97,:101`); it never accepts an `expandedToDescriptor(tpl,keys)` multisig descriptor. So T6b needs a NEW render path that (a) calls `expandedToDescriptor(tpl, keys)`, (b) on `expandTemplateOnly`/`expandUnsupported` shows display-only with NO `address.*` call, (c) on `expandOK` calls `address.Receive/Change`. The faithful-or-refuse *logic* is sound (`gui/md1_expand.go:32-78` returns a nil descriptor for non-sortedmulti, so a wrong-address verify is structurally impossible), but the spec's §3 NET-NEW list omits this render path and names a non-reusable function. Enumerate it and add the §5.5 acceptance test against the real multisig address.

## Minor (nice-to-have; can defer to FOLLOWUPS)

- **m-1 — `mk.Encode(Card{…})` omits `Network`.** §2 line 18 lists `Card{Path, Fingerprint, Stubs, Xpub}` with no `Network`. The shipped single-sig path sets `Network: networkName(net)` (`gui/singlesig_derive.go:74`). `Network` is *not* serialized into the mk1 bytecode (`mk/encode.go:69-95` — it never appends Network) and `bundle.Verify` does not compare it (`bundle/verify.go:52-60`), so this is not a correctness/verify defect, but the decoded card's network *label* would be empty/wrong. Set `Network: "mainnet"` for consistency with the single-sig flagship.

- **m-2 — §3 says "8 enumerated sites" but there is a 9th `qaProgram` reference at `gui/gui.go:1606`.** `return startScreenAction{prog: qaProgram}, true` (the `FOREVERLAURA!` debug entry). It is *name-based*, so inserting `engraveMultisig` (shifting `qaProgram` 4→5) does not break it — but the §3 enumeration should note it as "name-based, no change needed" so the implementer doesn't either miss it or wrongly assume the "8 sites" list is exhaustive of all `qaProgram`/enum references. Benign, but worth a sentence to prevent lockstep-audit confusion.

- **m-3 — §3 should call out `layoutMainPlates`'s `panic("invalid page")` explicitly.** `gui/gui.go:1856-1861` is a `switch` with a panic default; a navigable `engraveMultisig` that is NOT added to the `case` panics on render. The spec cites `:1856` but does not flag it as a *panic* site (unlike the title switch, which fails open to a blank string). The nav-test `TestEngraveSingleSigProgramNavigable` already documents this ("layoutMainPlates must have a case for it or it panics", `singlesig_program_test.go:11`). Surface it in the spec so the implementer treats it as mandatory, not cosmetic.

- **m-4 — Ambiguous-case wording is loose but the rule is sound.** §2 says "ambiguous (≥2, reused key) → engrave the FIRST-by-index slot's mk1 … policy+stub identical across slots, only mk1 Path differs." Two slots can both match ONLY if the seed legitimately appears at two *different* origin paths (since `match_k ≡ derive(seed, origin_k)==xpub_k`; identical derived keys at distinct origins). So the differing field is indeed only the mk1 Path, and first-by-index is deterministic — the rule is correct. But "reused key" is imprecise (it's "same seed at two cosigner slots"); tighten the wording and add the §5.1 test (key at @0 and @2 with distinct origins) exactly as the gate already promises. No behavior change required.

## Verified-correct (checked against source, confirmed sound)

- **D14 canonical-pair semantics (I-1 invariant).** `ExpandedKey.Xpub [65]byte` is chainCode[0:32]‖compressedPubkey[32:65] (`md/expand.go:62`; byte-split asserted at `md/expand_test.go:162-165`; existing slicing at `gui/md1_expand.go:65-66`). `decodeXpubBytes` returns the network-agnostic `(chainCode[32], compressedPubkey[33], parentFP)` and refuses xprv (`gui/singlesig_derive.go:99-122`). Comparing on this pair (not base58) is correct: the supplied xpub's version/depth/parentFP metadata differs, so base58 would spuriously fail; derivation ignores depth/childnum (noted at `md/expand.go:79-80`). Refuse-on-zero is the right wrong-wallet guard.
- **`deriveAccountXpub` derives at an ARBITRARY path and scrubs internally** (`gui/derive.go:19,33-52`): per-call it makes a fresh `seed` with `defer wipeBytes(seed)`, zeroes master + every intermediate, neuters, and (R0-C1 note `:46-52`) serializes the xpub BEFORE zeroing the final key. Calling it N times in the cross-match loop is leak-safe; the only secret living across the loop is the mnemonic `[]Word`, scrubbed by the flow's top-level `defer` (mirrors `gui/singlesig.go:41-45`). The §4 scrub spine holds across match / no-match / abort paths.
- **Full-policy gate (I-3).** `ExpandedKey.XpubPresent` exists (`md/expand.go:63`); the repo's `wsh_sortedmulti` golden is template-only and surfaces `XpubPresent=false` (`md/expand_test.go:100-102`), while a pubkey-bearing descriptor surfaces `XpubPresent=true` (`:143-166`). The "every slot XpubPresent or refuse" gate is load-bearing and well-founded.
- **Stub binding (I-4).** `md.WalletPolicyIDStubChunks(strs)` reassembles the SUPPLIED md1 strings and returns top-4 of the policy id (`md/walletpolicyid.go:124-134`) — bound to the supplied bytes, not a template-only/recomputed id. `mk.Encode(mk.Card{Network,Path,Fingerprint,Stubs [][4]byte,Xpub})` matches (`mk/encode.go:38`, `mk/mk.go:132-139`).
- **Verbatim engrave (I-2).** `bundleEngrave` → `bundlePlatePlan` engraves `c.strings` UNMODIFIED per plate (`gui/bundle_flow.go:303-318,327-357`); no re-encode. Engrave card order full=[ms1,mk1,md1] / watch-only=[mk1,md1]+reminder confirmed (`gui/singlesig_engrave.go:20-45`, reminder gate `gui/bundle_flow.go:364-380`).
- **`bundle.Verify` reuse (I-5).** Verifies stub-binding both sides, mk1 fp/xpub/path, md1 exact-string, ms1 recovered-entropy, with the watch-only both-empty-ms1 skip and a one-sided-ms1 presence-mismatch error (`bundle/verify.go:32-99`). It never inspects other cosigner slots — for the user's leg, derived.MD1 and readback.MD1 are both the supplied md1 verbatim, so the exact-string leg passes. Sound for "user's slot only."
- **Faithful-or-refuse restore (I-6).** `expandedToDescriptor` returns a nil descriptor for non-sortedmulti (`expandUnsupported`/`expandTemplateOnly`, `gui/md1_expand.go:36-49`), and only sortedmulti→P2WSH/P2SH/P2SH_P2WSH yields a descriptor (`:82-115`); `address.addressAt` handles SortedMulti incl. the P2SH_P2WSH nested case (`address/address.go:96-129`). A non-bip380 shape can never reach an address compare. (Logic sound; render path is I-3.)
- **`EncodeMS1` + `m.Valid()` gate.** `EncodeMS1(entropy)` (`codex32/msencode.go:17`) and the panic-on-invalid `Entropy()` (`bip39/bip39.go:158-161`) confirm the "gate `m.Valid()` first" requirement is real and load-bearing.
- **Program enum + lockstep.** Enum (`gui/gui.go:147-153`), dispatch (`:1491-1503`), left/right wrap with `engraveSingleSig` as the hardcoded bound (`:1636-1646`), title switch (`:1663-1672`), `npage`/`layoutMainPlates`/`npages` all keyed on `int(engraveSingleSig)+1`/the switch (`:1846,1856,1865`) — all 8 sites confirmed; all three nav-tests hardcode `engraveSingleSig` as the wrap upper bound (`singlesig_program_test.go:52-54`, `bundle_program_test.go:44-54`, `derive_xpub_program_test.go:31-32`) and MUST move. `qaProgram` stays non-navigable (no carousel case). The lockstep enumeration is complete (modulo m-2/m-3 annotations).
- **Mainnet-only coherence.** The cross-match is network-agnostic (chainCode+pubkey); `expandedToDescriptor` pins `&chaincfg.MainNetParams` (`gui/md1_expand.go:61`); consistent with the T6a full/watch-only branch (`gui/singlesig.go:63-95`, `bundle/verify.go:74-79`).

## Bottom line
The spec's security spine is correct: the D14 canonical-pair cross-match, refuse-on-zero, typed-only seed, per-leg scrub across the loop, verbatim engrave, watch-only-aware `bundle.Verify`, and faithful-or-refuse restore are all verified sound against the real `072461a` source — **zero Critical**. It is **not GREEN** on three Important items, all about *completeness of the cited reuse surface*, not about the security model: (I-1) the supply→single-cardMD1 extraction is unspecified and its named reuse (`singleSigReadbackCards`) actually rejects the valid one-md1/zero-mk1 case; (I-2) the normative D14 compare is written in a non-compiling `[N]byte == []byte` form for the one Critical-class primitive; (I-3) the multisig restore-doc render path is net-new but unenumerated and the cited `restoreDocFlow` is single-sig-only. Fold these three (define the md1-only extractor with its 0/≥2/stray-mk1 failure modes; rewrite the compare as `bytes.Equal(cc[:],…)`; enumerate the new multisig restore render path in §3 NET-NEW + §5.5), plus the four Minors, then re-dispatch. With I-1..I-3 closed this is ready for the implementation-plan phase.
