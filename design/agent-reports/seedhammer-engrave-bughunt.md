## Confirmed bugs

### 1. [CRITICAL] uint underflow on move-knot progress accounting in SafePointer.Progress
- **Location:** `engrave/engrave.go:1467-1470` (dim: engrave-geometry) — verifier confidence: high; filed high -> adjusted critical
- **Bug:** The advance loop only guards the early `break` for engrave knots: `if k.Engrave && s.progress < k.T { break }`. For a NON-engrave (move) knot it falls through to the unconditional `s.progress -= k.T` (line 1470). `s.progress` is a uint. The value fed to Progress() comes (via gui/engraver.go:160 -> splineResumer.Knot -> Driver.Knot 'completed') from the stepper driver, which only reports COMPLETED WHOLE WORDS (stepper.go:118-119, stepsPerWord = 32/5 = 6); a partial word is withheld until the next knot or Flush. Therefore the per-knot 'completed' reported to Progress can lag the knot's own duration k.T, so s.progress can be < k.T exactly when the next history knot to retire is a move knot. The subtraction then wraps to ~2^32.
- **Trigger + consequence:** Resume path (SafePointer.Resume builds leading non-engrave move knots, engrave.go:1455) or any plan whose first retired history knot is a move with T (in ticks) larger than the steps the driver has flushed so far. Example: first move knot T=1000, driver flushes only 996 steps (partial word held back) -> s.progress=996, knot is a move so no break, s.progress -= 1000 -> 4294966292. The loop then sees a gigantic progress, retires ALL remaining history knots (no engrave break fires until it wraps), copies/truncates s.history wrongly, and sets s.safePoint to the wrong control point. On a subsequent power-loss/resume the engraver moves to a wrong 'safe point' and replays/skips geometry -> misaligned engraving, possible head crash into already-engraved area, ruined plate.
- **Suggested fix:** Make the subtraction safe and the guard symmetric: stop (break) whenever s.progress < k.T regardless of Engrave, or clamp with `if s.progress < k.T { break }` before line 1470; alternatively `s.progress = saturatingSub(s.progress, k.T)` only after confirming the knot is fully elapsed. Reconcile the units: Progress receives driver step counts while k.T is in ticks; ensure they are the same unit or convert.
- **Verifier reasoning:** Confirmed at engrave/engrave.go:1467-1470. The break guard `if k.Engrave && s.progress < k.T` only protects engrave knots; a move knot (Engrave==false) falls through to the unconditional `s.progress -= k.T` where s.progress is uint (engrave.go:1448). I verified unit equivalence: k.T (ticks) == pio steps for the knot (bspline.Segment.Knot returns dt3 as ticks -> bezier.Interpolator.Segment(c,ticks) -> fraction produces exactly that many steps; bezier/bezier.go:86-104,53-67), so progress and k.T are comparable. The lag is structural: stepper.Driver.flush reports only whole flushed words (stepper/stepper.go:116-125, stepsPerWord=6) and the real mjolnir2.Device.Write returns DMA-consumed words which can be 0 when Driver.Knot returns (driver/mjolnir2/mjolnir2.go:217-273). Driving the real stepper.Driver over a real PlanEngraving plan, the first move knot (Engrave=false,T=10667) reports completed=0 under BOTH an eager whole-word device and a lazy-DMA device, i.e. completed<k.T on the leading move knot in the normal non-resume path. Feeding that into SafePointer.Progress reproduced the wrap: Progress(0) on a leading move knot with T=2771 sets s.progress=18446744073709540949 (~1.8e19). Decisively, the repo's OWN TestSafePointer already exhibits the wrap: tracing seed 42, after iteration 0 sp.progress=18446744073709549990; the test passes only because its assertions check postfix-matching, not the counter. Once wrapped, the engrave break (s.progress<k.T) can never fire, so the loop retires ALL history knots (engrave included) and selects the safePoint from corrupted state (lines 1473-1485). Real plans always begin with a move-to-start (verified first knot Engrave=false T=2771) and SafePointer.Resume prepends a leading non-engrave move (engrave.go:1455), so the trigger shape is the universal start of every engraving, not a rare resume edge.
- **Refutation considered (and rejected):** Strongest non-bug argument: the driver's cumulative `completed` catches up across later knots (e.g. eager device knot[2] reports ~knot[0]'s T), so over the whole run sum(progress)==sum(k.T); and the splineResumer clamps progress to >=0 (gui/engraver.go:213-214). One could argue the SafePointer only needs eventual consistency and the existing TestSafePointer passes. This refutation FAILS: the catch-up arrives only AFTER the leading move knot has already been retired at line 1470 with s.progress<k.T, and tracing shows the wrap happens on iteration 0 of the very test cited as evidence of correctness — the test passes by luck of its weak assertions, not because the wrap is avoided.

### 2. [HIGH] EngraveSeedString panics (uncaught) for codex32 strings whose QR exceeds version dim 33
- **Location:** `backup/backup.go:75-87 (panic originates via line 77 qr.Encode + line 81 engrave.ConstantQR)` (dim: backup-layout) — verifier confidence: high; filed critical -> adjusted high
- **Bug:** EngraveSeedString encodes the (uppercased) seed into a QR with qr.Encode(seed, qr.M) (line 77) and then calls engrave.ConstantQR(qrc) (line 81) with NO guard on the resulting QR version/size. The constant-time QR engraver only supports QR dimensions 21/25/29/33 (engrave.go bitmapForQRStatic / constantTimeQRModules). For any larger version it hits `default: panic("unsupported qr code version")` in engrave.go:399. Because ConstantQR is called eagerly, the panic propagates out of EngraveSeedString itself -- it is NOT returned as an error. The sole GUI caller backupSeedStringFlow (gui/gui.go:2078-2081) only checks `if err != nil { return }`, which a panic bypasses entirely. On the RP2350 firmware there is no watchdog, so the panic crashes/hangs the controller mid-flow with no recovery.
- **Trigger + consequence:** Empirically reproduced with real, valid codex32 strings: a 93-char short-code (e.g. 44-byte payload `ms10testsqqrsu9guyv4rzwplgex4gkmzd9c8wl593jfe4gdg47mtm3xt6tv7pelw7h7qxzs3rq0jvtg3ye6xggmhcl92`) encodes to QR dim 37, and 125-127-char long codes encode to dim 41. codex32.New accepts lengths 48-93 and 125-127 (codex32.go:101-104), and this repo's entire purpose is engraving md1/mk1/ms1 constellation strings via exactly this codex32 SeedString path (gui/codex32_polish.go:228 -> backupSeedStringFlow). Feeding any such longer-but-valid codex32 string at the engrave step panics: confirmed output `len=93 PANIC (uncaught): unsupported qr code version`, while len<=88 returns normally. Consequence: device panic/hang (DoS), no plate produced, user stuck.
- **Suggested fix:** In EngraveSeedString, after qr.Encode, guard the version before calling ConstantQR: reject QR dims outside {21,25,29,33} by returning an error (e.g. `if qrc.Size > 33 { return nil, errors.New("seed too long to engrave QR") }`), so the existing err!=nil path handles it gracefully. Alternatively make engrave.ConstantQR/bitmapForQRStatic return an error instead of panicking for unsupported versions. Either way EngraveSeedString must never panic on caller-reachable input length.
- **Verifier reasoning:** Confirmed by reading the code and by empirical reproduction with the repo's exact dependencies. engrave/engrave.go:384-400 panics ("unsupported qr code version") for any QR dim not in {21,25,29,33}; ConstantQR (engrave.go:406-410) calls bitmapForQRStatic eagerly. backup/backup.go:75-87 (EngraveSeedString) calls qr.Encode(seed, qr.M) then engrave.ConstantQR(qrc) with no version/size guard, so the panic propagates out of the function rather than being returned as the err it checks. The sole GUI caller backupSeedStringFlow (gui/gui.go:2076-2092) only does `if err != nil { return }`, which a panic bypasses; there is no recover() anywhere in gui/backup/engrave. The path is reachable: gui/codex32_polish.go:259-266 accepts any string passing codex32.New, and codex32StatusLine + the dispatch explicitly support short (48-93) and long (125-127) codex32 strings; line 228 builds backup.SeedString{Seed: scan.String()} and calls the flow with no QR-version cap. codex32/codex32.go:41-44,101-107 confirms accepted lengths. Empirically, using qr v0.3.2 and the repo module: the claim's exact valid 93-char short code -> QR dim 37 -> 'PANIC: unsupported qr code version'; a real valid 127-char long vector (ms100c8vsm32..., BIP-93 reference) -> dim 41 -> same panic, both via engrave.ConstantQR and via EngraveSeedString. Crossover for alphanumeric payloads: <=90 -> dim 33 (OK), >=93 -> dim 37 (panic), 127 -> dim 41.
- **Refutation considered (and rejected):** Strongest non-bug argument: most BIP-93 test vectors are 48-74 chars (dim <=33), and a minimal codex32 share can be as short as 48 chars, so a casual user may never trigger it. Also the claim's stated boundary is slightly wrong (it says len<=88 returns normally, but the actual short-code crossover is between 90 and 93). However neither saves it: the GUI intentionally accepts the 125-127 long window and short codes up to 93, all of which exceed dim 33, and the panic is uncaught. So the refutation fails.

### 3. [HIGH] frontSideSeed column-2 top and bottom overlap for 33-word SLIP39 mnemonics, garbling the plate
- **Location:** `backup/backup.go:171-213 (layout constants 171-176; top col2 195-197; bottom col2 208-213)` (dim: backup-layout) — verifier confidence: high; filed high -> adjusted high
- **Bug:** The seed-plate column layout is hardcoded for <=24-word BIP39 mnemonics: maxCol1=16, maxCol2=4 (lines 172-173). Column 1 holds words 0..endCol1 (<=16), column-2 TOP holds words endCol1..endCol2 = words 16..min(20,N) (lines 195-197), and column-2 BOTTOM holds words endCol2..N anchored to the bottom of column 1 at y=(plateDims.Y+col1Height)/2 - (N-endCol2)*pfs (lines 210-212). There is no check that the bottom block fits in the space remaining below the top block. For N=33 the bottom block has 13 rows and its top edge rises ABOVE the bottom edge of the top block, so the two text blocks collide. The post-plan bounds check in gui.toPlate only rejects off-plate geometry (ErrTooLarge); it does NOT detect on-plate overlap, so this passes validation and is engraved.
- **Trigger + consequence:** Reproduced the exact layout math (faithful to lines 195-212) at the production scale (mm=6400 steps/mm, pfs=Params.F(4.1)). For N=33 (256-bit SLIP-39 share -- a selectable input: slip39LengthPick returns {20,23,27,30,33}, gui/slip39_polish.go:55): column-2 TOP rows (words 16-19) span y in [9.70,26.10]mm while column-2 BOTTOM rows (words 20-32) span y in [22.00,75.30]mm -- a ~4.1mm overlap, i.e. words 16-19 are engraved directly on top of words 20-21. N in {20,23,27,30} do not overlap; only 33 does. Consequence: a physically RUINED, unreadable seed-backup plate for any 33-word SLIP-39 share (the highest-security SLIP-39 option), with the overlapping region misengraved.
- **Suggested fix:** Either (a) widen the layout to support >24 words (e.g. compute per-column capacity from available height and add a third column / smaller font when N is large), or (b) at minimum guard against overlap: if (N-endCol2)*pfs would push the bottom block's top above the top block's bottom, return an error so the caller surfaces 'seed too long for plate' instead of engraving garbage. Add an explicit max-word-count assertion covering the 33-word case.
- **Verifier reasoning:** Confirmed the bug end-to-end in the actual code at HEAD f907eea.

LAYOUT MATH (backup/backup.go:171-213): maxCol1=16, maxCol2=4 (lines 172-173). For N=33: endCol1=min(16,33)=16, col1Height=pfs*16, endCol2=min(20,33)=20. Col2-TOP holds words 16..20 anchored at (plateY-col1Height)/2 (lines 195-197). Col2-BOTTOM holds words 20..33 (13 rows) anchored at (plateY+col1Height)/2 - (33-20)*pfs (lines 210-212). I reproduced the exact integer math (Params.F = round(v*mm), engrave/engrave.go:46-47; mm=6400 from platform_sh2.go:181, fullSteps=200/mmPerRev=8*Microsteps=256; pfs=F(4.1)=4.1mm; plateY=F(85)=85mm). Result: TOP block y=[9.70,26.10]mm, BOTTOM block y=[22.00,75.30]mm -> 4.10mm overlap (exactly one row pfs). N in {20,23,24,27,30} all yield gaps; ONLY 33 overlaps. The 16+4+4=24 column scheme was sized for BIP-39's 24-word max.

REACHABILITY: slip39/share.go ParseShare accepts word count 33 (line 86, case 20,23,27,30,33) and stores all 33 canonical words in Share.Mnemonic (canonical := make([]string, len(fields)), line 91; Mnemonic: canonical, line 128). 33-word = 256-bit master secret, a standard SLIP-39 form (slip39/vectors_test.go:111 '256-bit/33-word' vector idx 35). User flow: scanned share -> confirmSLIP39Flow can return slip39Engrave (gui/slip39_polish.go:126, a primary button) -> engraveSLIP39Verbatim (lines 379 and 426) -> backup.Seed{Mnemonic: scan.Mnemonic} (lines 488-489) -> EngraveSeed (backup.go:71) -> frontSideSeed (backup.go:161). No truncation, no multi-plate split; all 33 words go on one 85mm plate.

GUARD DOES NOT CATCH IT: toPlate (gui/gui.go:2817-2830) only checks attrs.Bounds.In(plate minus safetyMargin) and returns ErrTooLarge if the overall bounding box exceeds the plate (line 2823). Two overlapping text blocks both lie within the plate bounds, so this passes. The claim's characterization is exactly correct.

TEST GAP: backup/backup_test.go exercises only 12- and 24-word BIP-39 seeds (lines 114, 118), never >24, so the latent overlap was never caught.
- **Refutation considered (and rejected):** The strongest refutation would be that a 33-word share never reaches the verbatim single-plate path - e.g. if it were always routed to SLIP-39 Recover (which engraves the recovered BIP-39 master seed, max 24 words, on the native plate) or split across multiple plates. But verbatim engraving is an explicit user choice: slip39Engrave is a primary confirm button (slip39_polish.go:126) and the recovered-'Engrave shares' arm (line 426) also calls engraveSLIP39Verbatim, both feeding the full 33-word Mnemonic into the same single-plate layout with no truncation. A second refutation - that 33 is not a producible word count - fails: it is a standard SLIP-39 256-bit form with a test vector. No upstream guard, TinyGo specific, or input bound prevents it.
