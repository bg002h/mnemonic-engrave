# R0 GATE REVIEW — SPEC_seedhammer_T2c_md1_decode.md (commit ef6f5e8)

Reviewed against authoritative source `descriptor-mnemonic/crates/md-codec` **@ 0.36.0** (`Cargo.toml` version confirmed = `0.36.0`; tree at `c85cd49`) and fork `seedhammer` HEAD `2fed9b6` (matches the spec's stated base). All file:line citations below are independently verified, not taken from the spec/recon.

## Verification Results (per numbered fact)

1. **BCH alignment — CONFIRMED.** `MD_REGULAR_CONST = 0x0815c07747a3392e7` (`bch.rs:17`), `POLYMOD_INIT = 0x23181b3` (`bch.rs:32`). Regular code only, 13-symbol checksum (`codex32.rs:18`, `REGULAR_CHECKSUM_SYMBOLS = 13`), no error correction in the verify path (`bch_verify_regular` `bch.rs:89-96`; `unwrap_string` `codex32.rs:113-161`). The spec's §2.4 "decode clean codewords only" is faithful. (Note: `chunk.rs` DOES contain `decode_with_correction` — BCH error correction exists in 0.36.0 — but the spec correctly scopes it OUT and targets the no-correction `reassemble`/`decode_md1_string` path.)

2. **String layer — CONFIRMED.** Single header = 5 bits, bit4 `divergent_paths`, bits3..0 `version=4` (`header.rs:30-49`, `WF_REDESIGN_VERSION = 4` at :27). Chunked header = 37 bits = `[v3..v0](4) + chunked(1) + csid(20) + count-1(6) + index(6)` (`chunk.rs:36-85`). In-band dispatch on bit 0 of symbol 0 (`chunk.rs:606`, `lib.rs:10-11`). 64-chunk cap (`chunk.rs:250`, `SINGLE_STRING_PAYLOAD_BIT_LIMIT = 64*5` at :219). Per-chunk `37 + 8·|bytes|` (`chunk.rs:284`), byte-boundary split (`chunk.rs:262-268`). **Single-string IS the common case — empirically confirmed: 9/10 corpus vectors are single `md1…` strings; only `wsh_multi_chunked` is force-chunked.** I also empirically confirmed the dispatch: feeding a chunk-string to the single-payload `decode_md1_string` yields `WireVersionMismatch{got:9}` (the top-4-bits-version layout differs), so the bit-0 chunked-flag dispatch is genuinely load-bearing.

3. **Symbol-aligned bit count — CONFIRMED.** `unwrap_string` returns `bit_count = 5 × data_symbols.len()` (`codex32.rs:157`), NOT `len*8`. N=3 breaking case is a real source test (`codex32.rs:190-206`); `reassemble` recovers `payload_byte_count = (symbol_aligned_bit_count - 37) / 8` (`chunk.rs:328`) and the comment calls out N=3, N=8 (`chunk.rs:316-318`). The spec invariant 2.3 is correct.
   - **Minor wording issue:** 2.3 says "≤4 trailing padding bits tolerated." That ≤4 is the *codex32-layer* symbol padding (`codex32.rs:111-112`). The *TLV-decode-layer* rollback tolerates ≤7 (`tlv.rs:296`). Two distinct budgets; the spec uses ≤4 in 2.3 and the recon F3 uses ≤7 for TLV — both correct but the spec should name them separately to avoid an implementer using one threshold for both.

4. **kiw formula — CONFIRMED (with a Go-portability note).** Source: `(32 - (n as u32).saturating_sub(1).leading_zeros())` at **both** `encode.rs:40` and `decode.rs:26` (lockstep). Verified the exact values the prompt asked for: n=1→0, n=2→1, n=3→2, n=4→2 (and source test `tree.rs:336,349` pins n=1→0, n=3→2). The spec writes `32 - (n-1).leading_zeros()`, which is correct on the valid domain n≥1. **Note for implementation (not a spec defect):** source uses `saturating_sub(1)`; the Go port must guard the `n-1` subtraction so a malformed `n=0` does not underflow — but n=0 is already rejected upstream (`KeyCountOutOfRange`, `origin_path.rs:111`), so this is defense-in-depth.

5. **AST / tags — CONFIRMED (one count typo).** 36-tag 6-bit table `tag.rs:100-135`; renderable-subset codes exact: Wpkh 0x00, Tr 0x01, Wsh 0x02, Sh 0x03, Pkh 0x04, Multi 0x06, SortedMulti 0x07, MultiA 0x08, SortedMultiA 0x09 (+ TapTree 0x05). `0x3F` extension prefix consumes a 4-bit subcode then rejects (`tag.rs:156-162`); reserved 0x24..0x3E → `TagOutOfRange`. `MAX_DECODE_DEPTH = 128` (`tree.rs:185`). Root-tag allow-list `{Sh,Wsh,Wpkh,Pkh,Tr}` (`decode.rs:36-44`). Multi-family wire `tag(6)+k-1(5)+n-1(5)+n×index(kiw)` (`tree.rs:244-254`). Tr is `is_nums(1) | [key_index(kiw) iff !is_nums] | has_tree(1) | [tree iff has_tree]` (`tree.rs:268-292`) — the variable-width case.
   - **DISCREPANCY (minor):** spec §3 and recon F3 say `Body` is an **8-variant** enum. Source `tree.rs:18-73` has **9 variants**: Children, Variable, MultiKeys, Tr, KeyArg, Hash256Body, Hash160Body, Timelock, Empty. Correct count is 9.

6. **Paths — CONFIRMED.** Origin path `depth(4) + components`, each `hardened(1) + LP4-ext varint` (`origin_path.rs:28-39,54-76`), `MAX_PATH_COMPONENTS = 15` (:43). `PathDecl` = `n-1(5)` + shared/divergent by header bit4 (`origin_path.rs:110-146`). Multipath `has(1) + [alt-count-2(3) + alts] + wildcard(1)`, alts 2..9 (`use_site_path.rs:80-116`). LP4-ext varint `[L:4][payload:L]`, L=15 continuation, cap 2²⁹-1 (`varint.rs:15-56`).
   - **DISCREPANCY (Important — see Findings):** "NO std-path dictionary (contrast mk1)" is only true at the **wire** layer. There IS a 5-shape **canonical-origin dictionary** at the **validator** layer (`canonical_origin.rs:45-79`: pkh→44'/0'/0', wpkh→84'/0'/0', tr-keyonly→86'/0'/0', wsh-multi→48'/0'/0'/2', sh-wsh-multi→48'/0'/0'/1') that `decode_payload` consults via `validate_explicit_origin_required` (`decode.rs:68`).

7. **TLV — CONFIRMED.** `TLV_FINGERPRINTS = 0x01` (4B each, `tlv.rs:13,407-411`), `TLV_PUBKEYS = 0x02` (65B = 32-byte chaincode‖33-byte compressed pubkey, `tlv.rs:14-16,29,422-426`). `@N`-indexed via `read_sparse_tlv_idx` (kiw-bit idx, range-checked `PlaceholderIndexOutOfRange`, ascending `OverrideOrderViolation`, `tlv.rs:317-333`). ≤7-bit padding tolerance + rollback (`tlv.rs:215-303`, threshold at :296). `bit_limit`-scoped body reads guard against inflation (`tlv.rs:359-376`).

8. **Representation gap — CONFIRMED.** `bip380.Descriptor.Type ∈ {Singlesig, SortedMulti}` (`bip380/bip380.go:90-95`), `Script ∈ {P2SH, P2SH_P2WSH, P2SH_P2WPKH, P2PKH, P2WSH, P2WPKH, P2TR}` (:58-67). No ordered `Multi`, no `MultiA`/`SortedMultiA`, no thresh/miniscript/taptree. Cannot represent the md1 AST → summary-over-projection is justified.

9. **Corpus — CONFIRMED, with a recon byte-string error (not in the spec).** MANIFEST = **10 vectors**, all `keys: &[]` (template-only), `test_vectors.rs:42-56`; per-vector `tests/vectors/<name>.{template,phrase.txt,bytes.hex,descriptor.json}` exist; no SHA/JSON-envelope pin (parity provenance = per-vector files). All 4 named vectors exist; I decoded all 10 via `md inspect` and confirmed templates/n/bytes/descriptor.json. **The spec §6 is correct** (it says "embed verbatim from md-codec," embeds no byte strings). **But the recon it cites is wrong:** recon F7 lists `wsh_multi_2of3` bytes as `2082001821842180` — that is actually `wsh_multi_chunked`'s payload (k=3). The true `wsh_multi_2of3` bytes are `2082001821822180` (k=2). The recon transposed the two vectors' payloads (differ only in the 5-bit `k` field: k=2 vs k=3). Since parity tests must embed verbatim from the live `bytes.hex`, this will self-correct *if* the implementer pulls from source and not from the recon — but the plan should explicitly forbid copying byte strings from the recon.

## Design Assessments

- **Renderable subset (§4.2) — mostly sound but incomplete on the canonical-origin interaction.** It correctly distinguishes Multi (ordered) vs SortedMulti, k-of-n, script type, and Tr keyspend-only-vs-refuse-taptree. `Sh(Wsh(<multi>))` vs `Sh(<multi>)` is genuinely distinguishable in the AST (nested `Wsh` child vs direct `MultiKeys` body). **However**, three §4.2 "renderable" shapes — `Sh(Multi)`, `Sh(SortedMulti)`, and `Sh(Wpkh(@k))` — have `canonical_origin == None`, so a *clean* wire encoding them with elided origins is **rejected at decode** by md-codec with `MissingExplicitOrigin` (I confirmed all three empirically). They are renderable only when carrying explicit per-`@N` origins. The spec must state that `Renderable=true` shapes outside the canonical-origin table only arrive with explicit origins, and the renderer must surface those origins (its `OriginPath` field already supports this).

- **Decode-must-be-complete + `Renderable=false` Template — the spec's framing is partly wrong (Important).** The spec §4.1 says the full AST is decoded even when `!Renderable`, returning a `Renderable=false` Template "with no policy claim." That is true for the bit cursor, BUT md-codec's `decode_payload` runs five post-decode validators (`decode.rs:56-69`): `validate_placeholder_usage`, `validate_multipath_consistency`, `validate_tap_script_tree`, `validate_explicit_origin_required` (canonical-origin-gated), `validate_xpub_bytes`. I confirmed empirically that a valid-bit-cursor complex shape like `wsh(and_v(v:pk(@0),older(144)))` with elided origin **decodes to an ERROR** (`MissingExplicitOrigin`), not to a `Renderable=false` Template. So `Renderable=false` is reachable only for complex-but-validator-passing wires (e.g. those carrying explicit origins, or `tr(@0,taptree)`/`sh(sortedmulti)` *with* explicit origins). The spec must (a) distinguish "valid-but-complex → Renderable=false Template" from "invalid → decode error," and (b) decide whether the Go port replicates the full validator suite (required for invariant 2.1 byte-for-byte / 2.5 reject-set faithfulness) or deliberately narrows it (then 2.1/2.5 must be re-scoped).

- **Sub-scope (decode-core + template summary; defer xpub/bip380/address to T2c-b) — sound and standalone-testable.** The roadmap goal ("human-readable BIP-388 template") justifies summary-over-projection given the representation gap (Fact 8). All 10 corpus vectors are template-only, so the cycle is fully parity-testable without the xpub path.

- **Feasibility — confirmed.** The MSB-first bit reader (`bitstream.rs:88-209`) is trivial, dependency-free, bounds-checks every read (`read_bits` :128-133), and is straightforwardly TinyGo-safe in Go. Recursive `read_node` (`tree.rs:196-328`) ports directly with a depth counter. The gatherer generalization is real: `mk1Gatherer.offer` already keys off `mk.ParseHeader → {TotalChunks, ChunkSetID, ChunkIndex, Chunked}` (`gui/mk1_inspect.go:48-83`), and `NFCReader()==nil` makes the pure path unit-testable. `mdmkFlow`/`isMK`/`idx--` wiring (`gui/gui.go:1932-1967`) and `TestMdmkFlowMD1NoInspect` (`gui/mk1_inspect_test.go:140-156`) match §4.3 exactly.

## Findings

### CRITICAL
*(none)*

### IMPORTANT

- **I1 — The post-decode validator suite + canonical-origin dictionary are load-bearing for parity but absent from the spec.** Where: §3 ("validators" hand-waved), §4.1 (decode order ends "…→ validators"), §2.5 (reject set). Why it matters: `decode_payload` runs `validate_placeholder_usage`, `validate_multipath_consistency`, `validate_tap_script_tree`, `validate_explicit_origin_required`, `validate_xpub_bytes` (`decode.rs:56-69`), and the explicit-origin one consults a 5-shape canonical-origin table (`canonical_origin.rs:45-79`). The spec's invariant 2.1 ("byte-for-byte vs md-codec") and 2.5 (reject set) cannot be met without porting these. Concrete fix: enumerate the five validators and the canonical-origin table in §3/§4.1; add to 2.5's reject set the variants they raise — `MissingExplicitOrigin`, `PlaceholderNotReferenced`, `PlaceholderFirstOccurrenceOutOfOrder`, `MultipathAltCountMismatch`, `ForbiddenTapTreeLeaf`, `NUMSSentinelConflict` (and `InvalidXpubBytes`/`InvalidPresenceByte` even though xpub-expansion is deferred, because `validate_xpub_bytes` still runs on any Pubkeys TLV the spec says it parses for cursor correctness). Add corpus/constructed negatives for each.

- **I2 — `Renderable=false` Template vs decode-error are conflated.** Where: §4.1 ("the full AST is decoded even when not renderable; Decode returns a Template whose Renderable=false…"), §2.4. Why it matters: empirically, a complex shape with elided origins is a *decode error*, not a `Renderable=false` Template (confirmed: `wsh(and_v(...))` → `MissingExplicitOrigin`). Returning a Template there would be *less* faithful than md-codec (it would display a complex policy md-codec refuses to decode at all). Concrete fix: state that `Decode` returns `(Template, error)`; on any md-codec reject (incl. the I1 validators) it returns an error (GUI shows "cannot decode this card"), and `Renderable=false` is reserved for wires that fully decode AND validate but fall outside §4.2.

- **I3 — Three §4.2 "renderable" shapes are decode-rejected with elided origins.** Where: §4.2 bullet 2 (`Sh(<multi-family>)`, `Sh(Wpkh(@k))`). Why it matters: `canonical_origin` returns None for `sh(multi)`, `sh(sortedmulti)`, `sh(wpkh)` (confirmed empirically — all three → `MissingExplicitOrigin` without explicit origins); only `sh(wsh(<multi>))` and the single-key/wsh-multi/tr-keyonly forms are canonical. The spec should note these shapes only ever reach the renderer carrying explicit per-`@N` origins, and the renderer MUST display those origins (don't claim the canonical 49'/48' path). No code change to §4.2's *membership*, but the renderer contract and a parity vector (an explicit-origin `sh(multi)`) need adding.

### MINOR

- **M1 — `Body` variant count.** §3 / recon F3 say "8-variant"; source has 9 (`tree.rs:18-73`). Fix the number; the implementer builds this enum from it.
- **M2 — Padding-budget ambiguity.** §2.3's "≤4 trailing padding bits" is the codex32-symbol budget; the TLV-rollback budget is ≤7 (`tlv.rs:296`). Name both explicitly so neither is applied in the wrong layer.
- **M3 — Recon byte-string transposition.** The cited recon F7 lists `wsh_multi_2of3` bytes as the chunked vector's `…842180` (true value `…822180`). The spec §6 is fine (embeds nothing), but the plan should mandate "byte strings come verbatim from live `tests/vectors/*.bytes.hex`, never copied from the recon."
- **M4 — Single-string `ParseHeader` for the generalized gatherer.** §4.1's `ParseHeader(s) (Header, error)` and §4.3's `parseHeader → (setID, total, index, chunked, ok)` must return `chunked=false, total=1, index=0` for a non-chunked md1 (its 5-bit header has no csid/count/index fields). State this so the generalized gatherer's prime/complete logic handles a 1-string md1 set. (mk1 is always chunked, so this path is net-new.)
- **M5 — Tr `is_nums` variable width has no dedicated invariant.** The kiw-suppression-when-`is_nums` (`tree.rs:271-276`) is the single most cursor-fragile spot after kiw itself. Worth its own invariant or an explicit callout under 2.1, with a `tr(NUMS, multi_a(...))` parity vector — though note this is NOT in the 10-vector corpus (only `tr_keyonly` is), so it would be a constructed vector sourced from md-codec round-trip.

## Verdict

**NOT GREEN — 0C/3I**

No Critical findings — every wire-format fact the spec relies on is verified correct against md-codec 0.36.0, the kiw formula is exactly right, the bit reader is feasible, and the sub-scope is sound. But three Important findings must be folded before code: the spec under-specifies the post-decode validator suite + canonical-origin dictionary that invariants 2.1/2.5 depend on (I1), conflates the valid-but-complex `Renderable=false` Template with a decode error (I2), and lists three `Sh(...)` shapes as renderable without noting they are decode-rejected unless they carry explicit origins (I3). Fold I1–I3, persist this review to `design/agent-reports/seedhammer-T2c-md1-spec-review-R*.md`, and re-dispatch.
