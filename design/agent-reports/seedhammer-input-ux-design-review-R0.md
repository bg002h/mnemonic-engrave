## VERBATIM REVIEW OUTPUT

### CRITICAL findings

**C1. "Move accept-word to Button3" collides head-on with the keyboard's own Button3 binding.** The `Keyboard` widget consumes Button3 as a synonym for Center ("commit the selected key"): filter includes `ButtonFilter(Button3)` (`gui/gui.go:952`), handler is `case Center, Button3: k.rune(); return true` (`gui/gui.go:1009-1011`). In `inputWordsFlow`, `kbd.Update(ctx)` runs every frame (`gui/gui.go:551`) and swallows Button3 before any `okBtn` bound to Button3 sees it. So Part A.3 is not render-only — it's a behavioral conflict. Resolve in the spec. Preferred: remove `Button3` from the keyboard filter and `case` (gui.go:952, 1009) so the keyboard commits only on Center, freeing Button3 globally for the screen primary action; apply consistently across `inputWordsFlow`/`inputCodex32Flow` (gui.go:626)/`inputSLIP39Flow` (gui.go:685), which all currently put OK on Button2. Call out that this removes a convenience binding. Alternative: accept the word via tap-on-predicted-box only (weaker; abandons the consistency goal).

**C2. Tap-to-accept / tap-the-predicted-box are real in firmware but NOT drivable by the current test harness.** Event model has `PointerEvent{Pos}` (`gui/event.go:90-94`), `Clickable` handles via `PointerFilter` (`gui/widget.go:70,79-83`) — so tap targets are implementable. BUT the host harness has only `click`/`press`/`runes` (`gui/event_test.go:42-76`); zero pointer-at-coordinate injection in any `*_test.go`. So the "TDD on host" claim cannot cover tap behaviors. Spec must either (a) add a `tap(r,tag,pos)` coordinate-injection helper (resolve tag→bounds in `op.Input(...).Clip(...)`, gui.go:1116, 2090) as an explicit work item, or (b) explicitly accept tap paths are validated only by button-path tests + manual on-device QA.

**C3. Part A + Part B in one cycle is over-scoped — decompose.** Part A is small/additive/low-risk. Part B introduces in one cycle: a 3-page keyboard mode-machine, a `preserveCase` mutation of a security-critical shared widget, masked display + timed reveal, double-entry confirm, secret-string lifetime concerns, a fingerprint-choice screen, and a passphrase thread-through. ≥3 independently-reviewable subsystems, two carrying real design risk. Bundling defeats the R0 gate philosophy and produces a PR that "reviews worse" — the exact failure mode of closed PR #34. Decompose (see closing section).

### IMPORTANT findings

**I1. The "user chooses which fingerprint to engrave" is safer than the brief fears, but for a reason the brief gets wrong — and that should change the framing.** The choice does NOT interact with descriptor/xpub/multisig derivation: the seed-backup path is standalone (`newInputFlow`→`engraveObjectFlow` gui.go:1313,1689; a `bip39.Mnemonic`→`backupWalletFlow`→`engraveSeed` gui.go:1691-1692,1806-1812). `engraveSeed` computes the fingerprint purely as a human-readable LABEL (`backup.Seed.MasterFingerprint` gui.go:472, rendered `%.8X` backup/backup.go:119-120); it feeds NO key derivation. Descriptors are a separate object (`*bip380.Descriptor` gui.go:1732) arriving pre-built over NFC/QR. So the choice can't corrupt a multisig descriptor. The REAL risk is narrower: a fingerprint's job is to let a recoverer confirm "this plate reconstructs the wallet I expect." Engraving the PASSPHRASE-derived fingerprint creates two hazards: (1) recovery-by-mismatch — a recoverer restoring the bare seed computes a non-matching fingerprint and may think the backup is corrupt; (2) passphrase-existence leak — engraving the passphrase fingerprint reveals a passphrase exists, eroding plausible deniability (the point of a duress/hidden wallet). Recommended: DEFAULT to the bare-seed fingerprint (matches today, zero regression, deniability-preserving); if the user opts into the passphrase fingerprint, require an explicit `ConfirmWarningScreen` (gui.go:215-340) stating both hazards; and RECONSIDER whether the choice is needed in MVP at all — since the verify screen already shows both fingerprints, the simplest safe MVP always engraves bare-seed and defers the toggle. The toggle is the riskiest UX in the design and delivers the least; cut it from the first passphrase slice.

**I2. The 3-page keyboard is feasible on the existing widget, but `preserveCase` + page state are more invasive than "add a mode key."** Widget is built around one immutable `keys [][]keyboardKey` grid computed once in `NewKeyboard` (gui.go:790-843); `rune()` hard-uppercases (gui.go:1036). Cleanest: build THREE pre-computed grids (abc/ABC/?123) at construction + active index; `Layout`/`Update`/`adjust`/`adjustCol`/`rune()` index the active grid (cleaner than mutating one grid because `adjust`'s nearest-key search and per-key `Clickable`/`op.Input` tags must stay stable). `preserveCase`: `rune()` stops `unicode.ToUpper` (gui.go:1036), emits the grid key's literal rune; abc/ABC lives in WHICH grid is active. Trap: `Layout` also force-uppercases the display glyph (`unicode.ToUpper(key.r)` gui.go:1127) and the rune-event matcher lower-cases input (`unicode.ToLower(e.Rune)` gui.go:1015) — both must become case-exact under `preserveCase` or QWERTY/hardware-keyboard input silently mis-cases the passphrase. Mode/page keys: model as special runes (like the `'⌫'` sentinel handled in `Valid`/`rune()` gui.go:933,1032), not new event types. D-pad across pages: after a page switch, re-center the cursor (cf. `Clear` gui.go:846-848); don't wrap Left/Right across pages. Keep the BIP-39 dimming mask (`updateValidBIP39Keys`, indexes `key.r-'a'` into a 32-bit mask gui.go:921-930,884) OUT of the passphrase path (free-text, no dimming). ~150-250 lines touching the most security-sensitive widget — own slice + own R0.

**I3. The last-word checksum shortlist math is correct; integration needs care.** Correct: checksum bits = `len(entropy)/4` (`checksum()` bip39.go:175); last word packs `Word(last)<<checkBits | checksum` (`ChecksumWord` bip39.go:182-186). 24 words → 3 free entropy bits + 8 checksum bits, all fixed → exactly 1 valid last word. 12 words → 7 free + 4 checksum → 2^7 = 128 valid. Helper: `bip39.LastWordCandidates(prefix Mnemonic) []Word`, placed next to `ChecksumWord`/`FixChecksum` (bip39.go:120-186), pure-Go unit-testable. CAUTION: 128 is NOT a one-tap shortlist — for 12-word it's a filtered candidate set (restrict the keyboard valid-key mask to the 128, reusing `updateValidKeys` gui.go:921, OR make the shortlist 24-word-only and leave 12-word as today). Decide explicitly. The last-word special-case triggers when `selected == len(mnemonic)-1`, routing past `completeBIP39Word` (gui.go:554,562); keep words 1..N−1 untouched. Does NOT eliminate the `mnemonic.Valid()` check (gui.go:1973) — a transcription error in words 1..N−1 still fails; the brief slightly over-claims "eliminates the invalid-seed error class."

**I4. The `nvalid` match-count surfacing is trivially sound.** `updateValidBIP39Keys` returns `nvalid`, in scope in `inputWordsFlow` (gui.go:552,869-893). Pure additive render with existing data source. Nit: guard against showing "2048 matches" on empty fragment — show only after ≥1 char.

**I5. "Word N of 24" title is render-only and trivially safe.** `selected`+`len(mnemonic)` in scope (gui.go:539,595); already rendered into the predicted-word box (`selected+1` gui.go:595). Moving into `layoutTitle` (gui.go:612) is pure presentation.

### MINOR findings

**M1. Go string zeroization for the passphrase is real-but-acceptable; document, don't over-engineer.** `bip39.MnemonicSeed(m, password string)` (bip39.go:189) → PBKDF2 copies into a byte slice (bip39.go:197); Go strings are immutable/GC-managed, not reliably zeroizable. On an air-gapped, single-purpose, power-cycled device with no swap/persistence and no exfil path (NFC read-only inbound, gui.go:406), transient RAM residue is low-severity; masking handles the realistic (shoulder-surfing) threat. Keep the `string` signature (no churn), avoid logging (there's a `log` import gui.go:11), document the no-zeroize limitation. Don't gold-plate.

**M2. The never-engraved invariant is structurally automatic, not something the code must enforce.** `engraveSeed` builds the plate from `m` (word indices) only (gui.go:454-479); `seedqr.QR(m)` encodes word indices (`%04d` seedqr.go:24-33), CompactQR encodes `m.Entropy()` (seedqr.go:37-41) — no passphrase channel; the fingerprint is a one-way 4-byte hash. Invariant holds as long as the passphrase `string` is never added to a `backup.Seed`/`SeedString`/`Paragraph` field. Spec job: a negative assertion + a test at the `engraveSeed` boundary that inputs never include the passphrase.

**M3. The optional "Skip / Add passphrase" preamble fits `ChoiceScreen` cleanly.** `ChoiceScreen` (gui.go:1165-1294) is a ready 2-choice picker (Button1=cancel/Button3=choose gui.go:1181-1182), already used by `newInputFlow` (gui.go:1869). `{"SKIP","ADD PASSPHRASE"}` after `SeedScreen.Confirm` returns true (gui.go:1996, in `backupWalletFlow` before `engraveSeed` gui.go:1809-1812) is idiomatic. Warning better as a separate `ConfirmWarningScreen` or the `Lead` text.

**M4. Double-entry confirm is straightforward but needs a defined mismatch UX.** Plain string compare is fine (local self-check, not an attacker oracle). On mismatch, reuse `ErrorScreen` (gui.go:198-213) "Passphrases don't match", return to entry. Define whether mismatch clears both entries or just the second.

**M5. Thread-through of `passphrase` touches three signatures — keep it mechanical.** `deriveMasterKey(m,net)` (gui.go:187) hardcodes `""` (gui.go:188); `masterFingerprintFor(m,net)` calls it (gui.go:483); `engraveSeed` calls `masterFingerprintFor` (gui.go:455); `SeedScreen.Confirm` calls `deriveMasterKey` for validation (gui.go:1989). Add `passphrase string` to all four — mechanical. The validation call (gui.go:1989) must validate with the BARE seed (passphrase ""), since `deriveMasterKey` never fails for a valid mnemonic regardless of passphrase (PBKDF2 → always 64 bytes, gui.go:190-194). Passphrase affects only the displayed/engraved fingerprint label, NOT the validity gate. Don't over-thread.

**M6. Testing — strong for logic, partial for UI.** Harness is good at what Part A + keyboard logic need: `runes`/`click` drive entry (cf. `TestInputSeedCodex32` codex32_input_test.go:21-45), `ExtractText` (gui_test.go:470) asserts on-screen text, `synctest` (gui_test.go:158) gives deterministic time for the reveal timer (reuse `ConfirmDelay` gui.go:241-260). Feasible host tests: `bip39.LastWordCandidates` unit test (len==1/128, each `m.Valid()`); passphrase keyboard `runes("AbC123!")` → assert mask + char count + case-preserved fragment (test hook returning `kbd.Fragment`); fingerprint screen (known mnemonic+passphrase → both `%.8X` via `ExtractText`); double-entry mismatch (two passphrases → error screen text). Gaps: tap behaviors need the C2 helper; masked-glyph + timed-reveal visuals only partially assertable (text extraction won't prove the • glyph unless the mask substitutes a known sentinel).

**M7. Pre-existing infra debt the spec should NOT absorb but be aware of.** `fadeClip` no-op (gui.go:521-524) — cosmetic, irrelevant unless a candidate-list lands. The three theme alphas identical `0x55` (recon §2.6) → dimmed/disabled keys look like active ones; passphrase mode keys would benefit from real hierarchy, but that's a separate theme fix — don't bundle. Note only.

### Recommended decomposition & sequencing (3 R0-gated slices, smallest-risk first)

**Slice 1 — BIP-39 entry polish (Part A, low risk, no security surface):** Word N of N title (I5); surface `nvalid` (I4, empty-guard); resolve Button3 collision (C1) by freeing Button3 from the keyboard across all three flows and moving OK to Button3 consistently; `bip39.LastWordCandidates` (I3) + wire the 24-word one-word case, decide 12-word explicitly; tap-to-accept only if the C2 pointer helper is added this slice, else defer.

**Slice 2 — Passphrase keyboard widget (Part B.b, isolated, security-sensitive):** three-grid page model + mode keys + `preserveCase` (I2) with case-exact display/rune fixes; masked display + char count + timed reveal; unit/text-extraction tests for case preservation + masking. Ship as a self-contained widget change, no crypto wiring yet.

**Slice 3 — Passphrase flow + verification (rest of Part B):** skip/add preamble via `ChoiceScreen` (M3) + warning; `inputPassphraseFlow` + double-entry with defined mismatch UX (M4); thread `passphrase` through derive/fingerprint/engrave leaving the validity gate on the bare seed (M5); both-fingerprints verify screen; DEFAULT to engraving the bare-seed fingerprint — treat the "which fingerprint" toggle (B.f) as a separate optional sub-step gated behind a `ConfirmWarningScreen` carrying both I1 hazards, OR cut from MVP (strongest scope recommendation); negative test at the `engraveSeed` boundary (M2).

**Overall recommendation:** proceed to spec, but split into the 3 slices, fix C1 in the design, resolve C2's test-helper decision explicitly, and default the engraved fingerprint to bare-seed (deferring/gating the toggle). With those four changes the design is upstream-attractive and matches the codebase's grain.
